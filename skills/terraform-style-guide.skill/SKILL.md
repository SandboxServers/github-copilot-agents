# Terraform Style Guide

Code is read more than written. Optimize for the reader. Every resource, variable, and output gets a description. Consistency beats cleverness. If `terraform plan` shows no changes on a second run, you wrote it right.

## File Organization

Standard files in every root module and reusable module:

```
# Root Module
main.tf           # Primary resources (or split by logical group)
variables.tf      # All variable declarations
outputs.tf        # All output declarations
providers.tf      # Provider configurations and required_providers
terraform.tf      # Terraform block (required_version, backend)
locals.tf         # All locals blocks
data.tf           # All data sources

# Reusable Module
main.tf           # Resources
variables.tf      # Variable declarations with validation
outputs.tf        # Module outputs
README.md         # Generated by terraform-docs
```

### When to Split main.tf

- When main.tf exceeds ~200 lines, split by logical group
- Name files after the resource group: `network.tf`, `compute.tf`, `identity.tf`, `monitoring.tf`
- Keep related resources together (NSG + rules + NIC association in `network.tf`)
- Never split a single resource across files

## Naming Conventions

### Resources and Data Sources

```hcl
# Pattern: resource "type" "descriptive_name"
# snake_case, descriptive, avoid redundancy with the resource type

# GOOD
resource "azurerm_resource_group" "main" {}
resource "azurerm_virtual_network" "hub" {}
resource "azurerm_subnet" "app_gateway" {}

# BAD — redundant with resource type
resource "azurerm_resource_group" "resource_group" {}
resource "azurerm_virtual_network" "vnet" {}

# BAD — too short, meaningless
resource "azurerm_subnet" "s1" {}

# One of a type → use "main" or "this"
# Multiple of a type → use descriptive names
```

### Variables

```hcl
# snake_case, descriptive, prefixed by domain in large modules
variable "location" {
  description = "The Azure region for all resources."
  type        = string
  default     = "eastus2"
}

variable "vnet_address_space" {
  description = "Address space for the virtual network in CIDR notation."
  type        = list(string)
}

# Object variables for related config
variable "app_service_config" {
  type = object({
    sku_name       = string
    worker_count   = optional(number, 1)
    zone_balancing = optional(bool, false)
    always_on      = optional(bool, true)
  })
}
```

### Locals

```hcl
locals {
  name_prefix = "${var.project}-${var.environment}"
  common_tags = {
    Environment = var.environment
    Project     = var.project
    ManagedBy   = "terraform"
    Repository  = var.repository_name
  }
}
```

### Outputs

```hcl
# Pattern: <resource_type>_<attribute>
output "resource_group_name" {
  description = "The name of the resource group."
  value       = azurerm_resource_group.main.name
}

output "key_vault_uri" {
  description = "The URI of the Key Vault."
  value       = azurerm_key_vault.main.vault_uri
  sensitive   = true
}
```

## Formatting Rules

- Run `terraform fmt` before every commit. Non-negotiable.
- Align `=` signs within a block (terraform fmt does this).
- One blank line between resource blocks.
- No blank lines inside a resource block (except before nested blocks).

### Argument Ordering Within a Resource

```hcl
resource "azurerm_linux_web_app" "main" {
  # 1. Meta-arguments (count, for_each)
  for_each = var.apps

  # 2. Required arguments (identifiers first)
  name                = "${local.name_prefix}-app"
  resource_group_name = azurerm_resource_group.main.name
  location            = azurerm_resource_group.main.location
  service_plan_id     = azurerm_service_plan.main.id

  # 3. Optional arguments
  https_only                    = true
  public_network_access_enabled = false

  # 4. Nested blocks
  site_config {
    always_on           = true
    minimum_tls_version = "1.2"
  }

  identity {
    type = "SystemAssigned"
  }

  # 5. Tags (always last before lifecycle)
  tags = local.common_tags

  # 6. Lifecycle (always absolute last)
  lifecycle {
    ignore_changes = [site_config[0].application_stack]
  }
}
```

## Variable Design Checklist

- [ ] Every variable has a `description`
- [ ] Every variable has a `type` — no untyped variables
- [ ] `validation` blocks for business rules (CIDR ranges, naming patterns, allowed values)
- [ ] `default` only when a sensible universal default exists
- [ ] `sensitive = true` for secrets (prefer Key Vault references over secret variables)
- [ ] `nullable = false` when null is not a valid value

## Comments

```hcl
# GOOD: Explains WHY, not WHAT
# This role assignment needs time_sleep because Azure AD
# propagation takes 30-60 seconds after principal creation
resource "time_sleep" "role_propagation" { ... }

# BAD: Restates the code
# Create a time sleep resource
resource "time_sleep" "role_propagation" { ... }
```

- Explain WHY, not WHAT — the code shows what
- Multi-line comments for complex decisions
- Every `catch` and workaround needs a comment with context

## What NOT to Do

| Anti-Pattern | Why It's Wrong |
|---|---|
| `count` when `for_each` with a map is clearer | count reindexes on changes, causing replacements |
| `-target` in automation | Debugging tool, not a deployment strategy |
| Hardcoded subscription/tenant/resource IDs | Use data sources or variables |
| `depends_on` when implicit dependencies exist | Masks the actual dependency graph |
| Provider configurations in reusable modules | Pass from root via `configuration_aliases` |
| `terraform.workspace` for environment logic | Use variables — workspaces are a state tool |
| `.tfstate` in source control | Never. Use remote backends. |
| `local-exec` for resource creation | Use proper resources or data sources |
| Missing `description` on variables/outputs | Reading code without descriptions is archaeology |

## PR Review Quick Check

Before approving Terraform code:

- [ ] `terraform fmt` applied (no formatting diffs)
- [ ] All variables have `description` and `type`
- [ ] Resources follow argument ordering convention
- [ ] Tags applied from `local.common_tags`
- [ ] No hardcoded IDs, secrets, or magic numbers
- [ ] `for_each` preferred over `count` where applicable
- [ ] Comments explain non-obvious decisions
- [ ] Sensitive values use `sensitive = true`
- [ ] Modules have meaningful outputs
- [ ] File organization follows standard layout
