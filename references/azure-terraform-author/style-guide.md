# Terraform Style Guide

## Philosophy

- Code is read more than it is written. Optimize for the reader.
- Every resource, variable, and output gets a description. No exceptions.
- Consistency beats cleverness. Follow patterns even when shortcuts exist.
- If `terraform plan` shows no changes on a second run, you wrote it right. If it doesn't, fix it.
- Modules are contracts. Treat variable interfaces like APIs — breaking changes need versioning.

## File Organization

Standard files in every root module and reusable module:

```
# Root Module
main.tf           # Primary resources (or split by logical group)
variables.tf      # All variable declarations
outputs.tf        # All output declarations
providers.tf      # Provider configurations and required_providers
terraform.tf      # Terraform block (required_version, backend) — some teams combine with providers.tf
locals.tf         # All locals blocks
data.tf           # All data sources
versions.tf       # Alternative name for terraform.tf (pick one, be consistent)

# Reusable Module
main.tf           # Resources
variables.tf      # Variable declarations with validation
outputs.tf        # Module outputs
README.md         # Generated by terraform-docs
```

### When to split main.tf

- When a single main.tf exceeds ~200 lines, split by logical group
- Name files after the resource group: `network.tf`, `compute.tf`, `identity.tf`, `monitoring.tf`
- Keep related resources together (e.g., NSG + NSG rules + NIC association in `network.tf`)
- Never split a single resource across files

## Naming Conventions

### Resources and Data Sources

```hcl
# Pattern: resource "type" "descriptive_name"
# Use snake_case, be descriptive, avoid redundancy with the resource type

# GOOD
resource "azurerm_resource_group" "main" {}
resource "azurerm_virtual_network" "hub" {}
resource "azurerm_subnet" "app_gateway" {}
resource "azurerm_key_vault" "workload" {}

# BAD — redundant with resource type
resource "azurerm_resource_group" "resource_group" {}
resource "azurerm_virtual_network" "vnet" {}

# BAD — too short, meaningless
resource "azurerm_subnet" "s1" {}

# When there's only one of a type, use "main" or "this"
resource "azurerm_resource_group" "main" {}

# When there are multiples, use descriptive names
resource "azurerm_subnet" "app_gateway" {}
resource "azurerm_subnet" "private_endpoints" {}
resource "azurerm_subnet" "workload" {}
```

### Variables

```hcl
# snake_case, descriptive, prefixed by domain when in large modules
variable "location" {
  description = "The Azure region for all resources."
  type        = string
  default     = "eastus2"
}

variable "vnet_address_space" {
  description = "Address space for the virtual network in CIDR notation."
  type        = list(string)
}

# Object variables — use for related config that travels together
variable "app_service_config" {
  description = "Configuration for the App Service."
  type = object({
    sku_name          = string
    worker_count      = optional(number, 1)
    zone_balancing    = optional(bool, false)
    always_on         = optional(bool, true)
  })
}
```

### Locals

```hcl
# Use locals for computed values, repeated expressions, and naming
locals {
  # Resource naming — keep naming logic in one place
  name_prefix = "${var.project}-${var.environment}"
  
  # Tags — always define in locals, apply everywhere
  common_tags = {
    Environment = var.environment
    Project     = var.project
    ManagedBy   = "terraform"
    Repository  = var.repository_name
  }
}
```

### Outputs

```hcl
# Pattern: <resource_type>_<attribute>
output "resource_group_name" {
  description = "The name of the resource group."
  value       = azurerm_resource_group.main.name
}

output "resource_group_id" {
  description = "The resource ID of the resource group."
  value       = azurerm_resource_group.main.id
}

# Sensitive outputs
output "key_vault_uri" {
  description = "The URI of the Key Vault."
  value       = azurerm_key_vault.main.vault_uri
  sensitive   = true
}
```

## Formatting Rules

- Run `terraform fmt` before every commit. Non-negotiable.
- Align `=` signs within a block (terraform fmt does this automatically).
- One blank line between resource blocks.
- No blank lines inside a resource block (except before nested blocks).
- Order arguments: required first, then optional, then nested blocks, then lifecycle.

### Argument Ordering Within a Resource

```hcl
resource "azurerm_linux_web_app" "main" {
  # 1. Meta-arguments
  count    = var.deploy_app ? 1 : 0
  for_each = var.apps

  # 2. Required arguments (identifiers first)
  name                = "${local.name_prefix}-app"
  resource_group_name = azurerm_resource_group.main.name
  location            = azurerm_resource_group.main.location
  service_plan_id     = azurerm_service_plan.main.id

  # 3. Optional arguments
  https_only                    = true
  public_network_access_enabled = false

  # 4. Nested blocks
  site_config {
    always_on         = true
    minimum_tls_version = "1.2"
  }

  identity {
    type = "SystemAssigned"
  }

  # 5. Tags (always last before lifecycle)
  tags = local.common_tags

  # 6. Lifecycle (always absolute last)
  lifecycle {
    ignore_changes = [site_config[0].application_stack]
  }
}
```

## Variable Design Principles

- Every variable has a `description`. Every. Single. One.
- Every variable has a `type`. No untyped variables.
- Use `validation` blocks for business rules (CIDR ranges, naming patterns, allowed values).
- Use `default` only when there's a sensible universal default. Don't default just to make `plan` easier.
- Use `sensitive = true` for secrets. But prefer Key Vault references over passing secrets as variables.
- Use `nullable = false` when null is not a valid value (most of the time).

## Comments

```hcl
# Single-line comments for brief context
# Explain WHY, not WHAT — the code shows what

# GOOD: This role assignment needs time_sleep because Azure AD
# propagation takes 30-60 seconds after principal creation
resource "time_sleep" "role_propagation" { ... }

# BAD: Create a time sleep resource
resource "time_sleep" "role_propagation" { ... }

# Multi-line for complex decisions
# The Key Vault uses RBAC authorization mode instead of access policies
# because RBAC provides finer-grained control and integrates with
# Azure's identity governance. Access policies are legacy.
```

## What Not To Do

- Don't use `count` when `for_each` with a map would be clearer and safer (count reindexes on changes)
- Don't use `-target` in automation — it's a debugging tool, not a deployment strategy
- Don't hardcode subscription IDs, tenant IDs, or resource IDs — use data sources or variables
- Don't use `depends_on` unless absolutely necessary — implicit dependencies are preferred
- Don't put provider configurations in reusable modules — pass them from root
- Don't use `terraform.workspace` for environment logic — use variables
- Don't store `.tfstate` files in source control. Ever.
- Don't use `local-exec` provisioners for resource creation — use proper resources or data sources
- Don't skip `description` on variables, outputs, or modules. Reading code without descriptions is archaeology.
