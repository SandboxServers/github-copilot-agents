# Module Architecture

## Module Types
- **Resource Modules**: Wrap a single Azure resource or tightly coupled group (e.g., VM + NIC + OS disk). Reusable building blocks.
- **Pattern Modules**: Compose multiple resource modules into an architecture pattern (e.g., hub-spoke network, web app + database + KV). Higher-level abstractions.
- **Root Modules**: The top-level configuration that calls modules and configures providers/state. Not designed for reuse.

## Azure Verified Modules (AVM) — Reference Only
Microsoft's official Terraform module library. AVM modules are over-engineered for most use cases — they include extensive abstractions, deep nesting, and opinionated patterns that add complexity without proportional value. **Reference AVM for patterns but write your own modules.**

### How to Use AVM
- **Study AVM source code** to learn patterns: variable design, validation, output structure, testing approaches
- **Do NOT use AVM modules directly** in production — they are heavyweight and hard to customize
- Browse the [AVM registry](https://registry.terraform.io/namespaces/Azure) for reference implementations

### Why We Write Custom
- **Simpler** — Our modules wrap exactly what we need, nothing more
- **Controllable** — We own the interface, versioning, and upgrade path
- **Debuggable** — Fewer abstraction layers when troubleshooting
- **Performant** — Smaller plan/apply surface, faster execution

## Module Source Patterns
```hcl
# 1. Terraform Registry (public or private) — preferred for versioned modules
module "network" {
  source  = "myorg/virtual-network/azurerm"
  version = "~> 2.0"
}

# 2. Git repository (with tag-based versioning)
module "network" {
  source = "git::https://github.com/org/terraform-modules.git//modules/virtual-network?ref=v2.1.0"
}

# 3. Local path (monorepo or development)
module "network" {
  source = "../../modules/virtual-network"
}
```

### Version Pinning Rules
- **Registry**: Always use `version = "~> X.Y"` — allows patch updates, blocks breaking changes
- **Git**: Always use `?ref=vX.Y.Z` — never `ref=main`
- **Local**: No version pinning — you control the source directly

## Module Variable Design

### Flat Variables (Simple Modules)
```hcl
variable "name" {
  description = "The name of the resource."
  type        = string
  nullable    = false
}

variable "location" {
  description = "The Azure region."
  type        = string
  nullable    = false
}

variable "resource_group_name" {
  description = "The name of the resource group."
  type        = string
  nullable    = false
}
```

### Object Variables with Optional Attributes
```hcl
variable "network_config" {
  description = "Virtual network configuration."
  type = object({
    address_space     = list(string)
    dns_servers       = optional(list(string), [])
    subnets = map(object({
      address_prefixes                          = list(string)
      service_endpoints                         = optional(list(string), [])
      private_endpoint_network_policies         = optional(string, "Enabled")
      delegation = optional(object({
        name    = string
        actions = list(string)
      }))
    }))
  })

  validation {
    condition     = length(var.network_config.address_space) > 0
    error_message = "At least one address space must be provided."
  }
}
```

### When to Use Object vs Flat Variables
| Use Flat Variables When | Use Object Variables When |
|---|---|
| Module has <8 inputs | Related settings travel together |
| Simple resource wrappers | Complex nested config (subnets, rules) |
| Inputs are independently configurable | Settings are interdependent |
| Root module interfaces | Pattern modules with structured config |

## Module Composition (Root Module Pattern)
```hcl
# Root module orchestrating multiple child modules
# main.tf (root)

module "resource_group" {
  source   = "../../modules/resource-group"
  name     = "${local.name_prefix}-rg"
  location = var.location
  tags     = local.common_tags
}

module "network" {
  source              = "../../modules/virtual-network"
  name                = "${local.name_prefix}-vnet"
  resource_group_name = module.resource_group.name
  location            = module.resource_group.location
  address_space       = var.vnet_address_space
  subnets             = var.subnets
  tags                = local.common_tags
}

module "key_vault" {
  source              = "../../modules/key-vault"
  name                = "${local.name_prefix}-kv"
  resource_group_name = module.resource_group.name
  location            = module.resource_group.location
  tenant_id           = data.azurerm_client_config.current.tenant_id
  subnet_id           = module.network.subnet_ids["private-endpoints"]
  tags                = local.common_tags
}
```

## Provider Configuration in Modules
```hcl
# RULE: Reusable modules MUST NOT configure providers
# They declare required providers but don't set configuration

# In module's main.tf or versions.tf:
terraform {
  required_providers {
    azurerm = {
      source  = "hashicorp/azurerm"
      version = ">= 4.0"    # Minimum version, not pinned
    }
  }
}

# Provider config lives ONLY in root modules
# For multi-subscription, pass providers explicitly:
module "hub_network" {
  source = "./modules/virtual-network"
  providers = {
    azurerm = azurerm.hub
  }
  # ...
}
```

```

```

## Module Breaking Changes
When you need to make breaking changes to a module:
1. Bump the major version (v1.x → v2.0)
2. Document changes in CHANGELOG.md
3. Provide a migration guide (old variable names → new)
4. Use `moved` blocks where possible to avoid state disruption
5. Never remove an output without a major version bump
6. Add `deprecated` to variable descriptions before removing them
