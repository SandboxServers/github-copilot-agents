## Retrospective Types

### Sprint Retrospectives
- Conducted at the end of each sprint
- Address issues affecting effectiveness, productivity, and quality
- Address elements affecting satisfaction and project flow
- Identify incomplete backlog items and preventive actions
- Output: 1-3 specific process changes for the next sprint

### Release Retrospectives
- Conducted after major releases or milestones
- Improve cross-team communications and internal practices
- Broader scope than sprint retros — cover the full release cycle
- Output: organizational-level improvements

### Operational Reviews
- Typically held monthly
- Include representatives from the whole value stream
- Use objective, quantitative data (velocity, cycle time, deployment frequency)
- Provoke discussions about dynamics affecting performance between teams
- Output: cross-team patterns and systemic improvements

### Engagement Retrospectives (Post-Delivery)
- Conducted after completing a client engagement or major project
- Full scope: architecture, cost, security, testing, delivery, timeline
- Compare planned vs actual on all dimensions
- Output: lessons learned document, organizational knowledge base updates, action items with owners

### Post-Incident Reviews (Postmortems)
- Conducted within 72 hours of incident closure
- All stakeholders: technical responders, business owners, senior management
- Structured root cause analysis (Five Whys, Fishbone, timeline analysis)
- Output: RCA document, improvement work items, updated runbooks
