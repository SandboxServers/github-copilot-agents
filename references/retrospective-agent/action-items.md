## Action Item Tracking

### Purpose
Retrospectives without tracked action items are just venting sessions. Every insight must
translate into a specific, owned, time-bound action to drive actual improvement.

### Action Item Format
Every action item must include:
- **Description** — specific, concrete statement of what will be done
- **Owner** — a named individual (not a team or role)
- **Due date** — a specific calendar date
- **Status** — Not Started, In Progress, Completed, Deferred, Abandoned
- **Category** — Process Improvement, Tooling, Training, Documentation, Architecture
- **Success criteria** — how we verify the action was completed and effective

### Quality Standards
**Weak action items** (vague, unmeasurable):
- "Improve communication between teams"
- "Be more careful with deployments"
- "Write better tests"
- "Review architecture more thoroughly"

**Strong action items** (specific, assignable, verifiable):
- "Add egress cost estimation line item to the architecture review template — Owner: Sarah — Due: Jan 15"
- "Add private endpoint validation to the infrastructure test suite — Owner: Marcus — Due: Jan 22"
- "Create runbook for DNS resolution failures in hybrid environments — Owner: Priya — Due: Feb 1"
- "Update pipeline template to include Bicep what-if validation step — Owner: James — Due: Jan 20"

### Categories
Organize action items to identify where improvement effort is concentrated:
- **Process improvement** — changes to how the team works (reviews, handoffs, approvals)
- **Tooling** — new tools, tool upgrades, automation, pipeline changes
- **Training** — skill development, knowledge sharing, documentation of expertise
- **Documentation** — runbooks, architecture decision records, onboarding materials
- **Architecture** — template updates, reference architecture changes, design pattern fixes

### Follow-Up Cadence
- Sprint retrospective action items: review at the next sprint retrospective
- Engagement action items: review at 30-day and 60-day checkpoints
- Systemic action items: review monthly at the organizational level
- Never let action items go unreviewed — missed follow-up signals that retros don't matter

### Escalation for Overdue Items
When action items are consistently overdue:
- First occurrence: discuss blockers and adjust the due date with documented reasoning
- Second occurrence: escalate to the team lead or engagement manager
- Third occurrence: re-evaluate whether the action is feasible or should be abandoned
- If abandoned: document why and whether the underlying issue has an alternative fix

### Closing the Loop
Completed action items must be validated, not just checked off:
- Did the change produce the intended improvement?
- Measure the outcome: fewer defects, faster deployments, reduced cost, fewer repeat findings
- Share results with the team to reinforce that retrospective action items matter
- If the action did not produce the expected result, treat it as new input for the next retro
