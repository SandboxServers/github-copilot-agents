# Best Practices

## Module File Structure
Every module must have this minimum layout:
```
modules/<resource>/
├── main.tf           # Resources
├── variables.tf      # Inputs with validation
├── outputs.tf        # Outputs — only what consumers need
├── versions.tf       # required_providers block
└── README.md         # Generated by terraform-docs
```
Add `locals.tf` only when computed values are needed. Add `data.tf` only when data sources are numerous.

## Variable Validation
Validate inputs at the boundary — don't let bad values propagate:
```hcl
variable "environment" {
  type = string
  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Environment must be dev, staging, or prod."
  }
}

variable "cidr_block" {
  type = string
  validation {
    condition     = can(cidrhost(var.cidr_block, 0))
    error_message = "Must be a valid CIDR block."
  }
}
```

## Testing with `terraform test`
Write HCL-based tests for every module. See [testing.md](testing.md) for full patterns.
```hcl
# tests/main.tftest.hcl
run "creates_resource_group" {
  command = plan
  assert {
    condition     = azurerm_resource_group.main.location == "eastus2"
    error_message = "Wrong location."
  }
}
```

## Lifecycle Rules
Use the right lifecycle rule for the situation:
- **`prevent_destroy`** — Critical resources (databases, Key Vaults, storage)
- **`create_before_destroy`** — Zero-downtime replacements (certs, DNS records)
- **`replace_triggered_by`** — Force replacement when a dependency changes
- **`ignore_changes`** — Properties managed outside Terraform (autoscaler, policy tags)

## State Management
- Always use remote backend with locking (Azure Storage + `use_azuread_auth`)
- Use directory-based separation over workspaces for environment isolation
- Partial backend config for CI/CD — pass secrets at `terraform init` time
- Never store secrets in state variables; use Key Vault references instead
- See [state-management.md](state-management.md) for full configuration

## Provider Version Constraints
```hcl
required_providers {
  azurerm = { source = "hashicorp/azurerm", version = "~> 4.0" }
  azapi   = { source = "Azure/azapi",       version = "~> 2.0" }
}
```
- `~>` allows minor/patch but blocks breaking major changes
- Always commit `.terraform.lock.hcl` for reproducible builds
- Update providers deliberately, not accidentally

## Tagging Strategy with `default_tags`
```hcl
provider "azurerm" {
  features {}
  subscription_id = var.subscription_id
}

locals {
  common_tags = {
    Project     = var.project
    Environment = var.environment
    ManagedBy   = "terraform"
    CostCenter  = var.cost_center
  }
}
```
Apply `tags = local.common_tags` to every resource. Merge with resource-specific tags via `merge(local.common_tags, { Role = "database" })`.

## Locals for DRY Code
```hcl
locals {
  name_prefix = "${var.project}-${var.environment}"
  location    = var.location
  # Computed values — don't repeat this expression in 10 resources
  subnet_ids  = { for s in azurerm_subnet.this : s.name => s.id }
}
```

## Output Design
Expose only what downstream consumers need. Don't dump every attribute:
```hcl
# GOOD — curated outputs
output "id" { value = azurerm_key_vault.main.id }
output "vault_uri" { value = azurerm_key_vault.main.vault_uri }
output "name" { value = azurerm_key_vault.main.name }

# BAD — leaking internal details
output "key_vault" { value = azurerm_key_vault.main }
```

## Conditional Resource Creation
```hcl
# Binary — use count
resource "azurerm_private_dns_zone" "this" {
  count = var.enable_private_dns ? 1 : 0
  name  = "privatelink.azurewebsites.net"
  # ...
}

# Collection — use for_each
resource "azurerm_subnet" "this" {
  for_each         = var.subnets
  name             = each.key
  address_prefixes = each.value.address_prefixes
  # ...
}
```

## `depends_on` as Last Resort
Terraform resolves most dependencies implicitly through resource references. Use `depends_on` only when there is no reference chain (e.g., RBAC role assignments that must exist before a resource can use them). Explicit `depends_on` disables parallelism for the dependent resource.
