# Error Handling

## Error Types

PowerShell has three distinct error types. Knowing which commands produce which is critical for correct handling.

| Type | Behavior | Generated By |
|------|----------|-------------|
| **Non-terminating** | Pipeline continues; next command runs. Only caught by `catch` when escalated via `-ErrorAction Stop`. | `Write-Error`, `$PSCmdlet.WriteError()`, most cmdlet runtime errors |
| **Statement-terminating** | Current statement stops; next statement in the scope runs. Caught by `catch` without escalation. | `$PSCmdlet.ThrowTerminatingError()`, .NET method exceptions, engine parse-time errors |
| **Script-terminating** | Unwinds the entire call stack until caught or script exits. | `throw`, escalated non-terminating errors in non-advanced contexts, parse errors |

## ErrorAction and $ErrorActionPreference

### The Asymmetry

```
-ErrorAction Stop on compiled cmdlet          → statement-terminating
-ErrorAction Stop on advanced function        → statement-terminating
-ErrorAction Stop on NON-advanced function    → script-terminating
$ErrorActionPreference = 'Stop' in advanced   → statement-terminating
$ErrorActionPreference = 'Stop' elsewhere     → script-terminating
```

- `-ErrorAction` overrides `$ErrorActionPreference` for **one command**, applies only to **non-terminating** errors.
- `$ErrorActionPreference` applies to **all commands in scope and child scopes**.
- `.ThrowTerminatingError()` ignores `-ErrorAction` entirely (except the `Break` value).
- `throw` can be suppressed by `$ErrorActionPreference = 'SilentlyContinue'` — counterintuitive but real.

### Scope Behavior

```powershell
$ErrorActionPreference = 'Stop'   # Script scope

function Invoke-Inner {
    # Inherits 'Stop' from parent scope
    Get-Item -Path 'C:\nonexistent'   # Now terminates
}

function Invoke-Isolated {
    $ErrorActionPreference = 'Continue'   # Local override
    Get-Item -Path 'C:\nonexistent'       # Non-terminating again
}
```

### Always Use -ErrorAction Stop in Try Blocks

```powershell
try {
    # Without -ErrorAction Stop, non-terminating errors SKIP the catch block
    Get-ChildItem -Path 'C:\no\such\path' -ErrorAction Stop
}
catch {
    Write-Warning "Caught: $($_.Exception.Message)"
}
```

## Try/Catch/Finally

### Typed Exception Catches (Most Specific First)

```powershell
function Get-SafeResource {
    [CmdletBinding()]
    param([string]$Name)

    try {
        $result = Get-AzResource -Name $Name -ErrorAction Stop
        return $result
    }
    catch [Microsoft.PowerShell.Commands.HttpResponseException] {
        Write-Warning "HTTP error for ${Name}: $($_.Exception.Message)"
        return $null
    }
    catch [System.Management.Automation.ItemNotFoundException] {
        Write-Warning "Resource not found: $Name"
        return $null
    }
    catch {
        # $_ (alias: $PSItem) is the ErrorRecord
        Write-Error "Unexpected error: $($_.Exception.Message)"
        throw   # Re-throw preserves original stack trace
    }
}
```

### Finally for Cleanup

```powershell
$connection = $null
try {
    $connection = Open-DatabaseConnection -Server $Server
    Invoke-Query -Connection $connection -Query $sql
}
catch {
    Write-Error "Query failed: $_"
    throw
}
finally {
    # Runs whether or not an error occurred
    if ($connection) {
        $connection.Close()
        $connection.Dispose()
    }
}
```

## $PSCmdlet.ThrowTerminatingError()

Use over bare `throw` inside advanced functions. Produces a **statement-terminating** error with proper origin information (command name, pipeline position) instead of a script-terminating error pointing at the `throw` line.

```powershell
function Get-ConfigFile {
    [CmdletBinding()]
    param(
        [Parameter(Mandatory)]
        [string]$Path
    )

    if (-not (Test-Path -Path $Path)) {
        $PSCmdlet.ThrowTerminatingError(
            [System.Management.Automation.ErrorRecord]::new(
                [System.IO.FileNotFoundException]::new("Could not find $Path"),
                'ConfigFile.NotFound',
                [System.Management.Automation.ErrorCategory]::OpenError,
                $Path
            )
        )
    }
    Get-Content -Path $Path -Raw | ConvertFrom-Json
}
```

### Re-throwing Caught Errors Properly

```powershell
catch {
    # WRONG — creates new ErrorRecord, loses original context:
    # throw $_

    # RIGHT — re-throws original ErrorRecord:
    $PSCmdlet.ThrowTerminatingError($_)
}
```

## Trap Statement

Traps catch errors at the **scope level** (function or script). They execute inline, then control resumes based on `continue` or `break`.

```powershell
function Invoke-Trapped {
    # Typed trap — catches only this exception type
    trap [System.IO.IOException] {
        Write-Warning "IO error trapped: $_"
        continue   # Suppress error, resume at next statement
    }

    # General trap — catches everything else
    trap {
        Write-Error "Unhandled error trapped: $_"
        break      # Stop execution, propagate error up
    }

    Get-Content -Path 'C:\missing.txt' -ErrorAction Stop
    Write-Host 'This runs if trap used continue'
}
```

**When to use trap vs try/catch:**
- `trap` — logging/auditing at function scope; you want a blanket handler without wrapping every statement.
- `try/catch` — targeted handling around specific operations; the standard choice for most code.

## -ErrorVariable Pattern

Collect errors without stopping execution. The `+` prefix **appends** to the variable.

```powershell
$importErrors = [System.Collections.ArrayList]::new()

foreach ($file in $csvFiles) {
    Import-Csv -Path $file -ErrorAction SilentlyContinue -ErrorVariable +importErrors |
        ForEach-Object { Process-Record $_ }
}

if ($importErrors.Count -gt 0) {
    Write-Warning "$($importErrors.Count) file(s) had import errors:"
    $importErrors | ForEach-Object { Write-Warning "  - $($_.Exception.Message)" }
}
```

## Native Command Error Handling

### $LASTEXITCODE and $?

```powershell
git clone https://github.com/org/repo.git 2>&1
if ($LASTEXITCODE -ne 0) {
    throw "git clone failed with exit code $LASTEXITCODE"
}
```

- `$LASTEXITCODE` — integer exit code from last native executable (`0` = success by convention).
- `$?` — `$true` if last operation succeeded. For native commands, reflects `$LASTEXITCODE -eq 0`.

### PS 7.4+ Native Error Integration

```powershell
# Opt in: native command non-zero exit codes generate errors
$PSNativeCommandUseErrorActionPreference = $true
$ErrorActionPreference = 'Stop'

try {
    terraform plan -out=tfplan
}
catch {
    Write-Error "Terraform failed (exit $LASTEXITCODE): $($_.Exception.Message)"
}
```

### Wrapper Pattern for Pre-7.4

```powershell
function Invoke-NativeCommand {
    [CmdletBinding()]
    param(
        [Parameter(Mandatory)]
        [string]$Command,
        [Parameter(ValueFromRemainingArguments)]
        [string[]]$Arguments
    )
    $output = & $Command @Arguments 2>&1
    if ($LASTEXITCODE -ne 0) {
        $PSCmdlet.ThrowTerminatingError(
            [System.Management.Automation.ErrorRecord]::new(
                [System.InvalidOperationException]::new("$Command exited with code $LASTEXITCODE"),
                'NativeCommand.Failed',
                [System.Management.Automation.ErrorCategory]::InvalidResult,
                $Command
            )
        )
    }
    $output
}
```

## Retry Pattern for Transient Errors

```powershell
function Invoke-WithRetry {
    [CmdletBinding()]
    param(
        [Parameter(Mandatory)]
        [scriptblock]$ScriptBlock,
        [int]$MaxRetries = 3,
        [int]$BaseDelaySeconds = 2
    )
    $attempt = 0
    do {
        $attempt++
        try {
            return & $ScriptBlock
        }
        catch {
            if ($attempt -ge $MaxRetries) {
                throw   # All retries exhausted
            }
            $delay = $BaseDelaySeconds * [math]::Pow(2, $attempt - 1)
            Write-Warning "Attempt $attempt failed: $($_.Exception.Message). Retrying in ${delay}s..."
            Start-Sleep -Seconds $delay
        }
    } while ($attempt -lt $MaxRetries)
}

# Usage
$response = Invoke-WithRetry -MaxRetries 5 -BaseDelaySeconds 1 -ScriptBlock {
    Invoke-RestMethod -Uri $apiEndpoint -ErrorAction Stop
}
```

## Error Handling in ForEach-Object -Parallel

Each parallel runspace has its own scope. `try/catch` works inside each runspace, but error collection must be thread-safe.

```powershell
$errors = [System.Collections.Concurrent.ConcurrentBag[object]]::new()
$config = $using:config   # Capture once via $using:

1..20 | ForEach-Object -Parallel {
    $bag = $using:errors
    try {
        $result = Invoke-RestMethod -Uri "$($using:config.BaseUrl)/item/$_" -ErrorAction Stop
        $result
    }
    catch {
        $bag.Add([PSCustomObject]@{
            Item      = $_
            Error     = $_.Exception.Message
            Timestamp = [datetime]::UtcNow
        })
    }
} -ThrottleLimit 5

if ($errors.Count -gt 0) {
    Write-Warning "$($errors.Count) parallel operations failed"
}
```

## Error Inspection

```powershell
# PS 7.0+ — richest error detail
Get-Error               # Last error, fully expanded
Get-Error -Newest 3     # Last 3 errors

# $Error automatic variable — ArrayList, most recent at [0]
$Error[0].Exception.GetType().FullName
$Error[0].Exception.Message
$Error[0].Exception.InnerException   # Nested/wrapped exceptions
$Error[0].ScriptStackTrace           # PowerShell call stack
$Error[0].InvocationInfo             # Command that threw
$Error[0].CategoryInfo               # ErrorCategory details
$Error.Clear()                       # Reset (useful in tests)
```
